01 — Context
Why we built this system
VUC operates across multiple verticals simultaneously. The problem was not a lack of ideas — it was that ideas were getting lost, projects lacked clear ownership, and founders were being pulled into operational details instead of strategic decisions.
The core problem we were solving
High idea generation, low conversion to execution. No structured tracking. Implicit project ownership. No meeting cadence that closed the loop from ideation to execution. Founders as the bottleneck on operational decisions. No separation between ideas, validated opportunities, and active execution.
The design answer
Execution is protected from the noise of ideation.
Ideas have their moment (Friday) and their place (Jira). Active projects have their moment (Monday) and their owner. VWC has its own lane (Tuesday morning). Mixing any of these three destroys team focus.
02 — Foundation
Core principles
Every rule in this system traces back to one of these principles. When in doubt, come back here.
Principle 1
If it is not in Jira, it does not exist.
Ideas discussed verbally and not captured are lost. Urgency stated verbally does not move tickets. Non-negotiable.
Principle 2
Every idea and project has one named owner.
Ownership by committee means no one is actually responsible. One owner per ticket, always.
Principle 3
Execution is protected from ideation.
Monday never discusses new ideas. Friday never makes decisions. Each meeting has a single purpose.
Principle 4
Urgency must be written, not stated.
If urgency is not written in the Jira ticket, it does not move. Urgency lives in the ticket, not in the room.
Principle 5
Decisions happen in designated meetings.
Strategic decisions belong in Pipeline Review, Incubation, or Strategic Review. Teams chat is for communication, not decisions.
Principle 6
VWC is a separate lane.
VWC has its own meeting, Jira board, four-pillar structure, and KPI reporting. Mining never enters VUC pipeline discussions.
03 — Core system
The idea-to-execution pipeline
Every idea follows the same path. One exception: if an MVP already exists, the idea skips Idea Pool and enters Incubation directly.
Friday
Idea Pool
Captured with owner + urgency. Not evaluated yet.
Tuesday PM
Under Review
4 requirements checked. Owner confirmed. Urgency written.
Bi-weekly
Incubation
5-min case. Scored on Fit × Feasibility.
Monthly
Decision Engine
Formally scored. Enters Priority Allocation.
Monday
Active Execution
In the Top 5. Weekly commitments and Eisenhower.
Why this many stages?
Each stage has a specific job. The filter gets progressively tighter — by the time we dedicate team capacity to something, it has already survived four rounds of evaluation. Idea Pool captures without judging. Under Review checks minimum viability. Incubation validates strategic fit before investing scoring time. The Decision Engine applies rigorous weighted scoring. Active Execution is protected — only scored and approved ideas reach Monday.
📌
MVP fast-track
If a project already has a working MVP, it skips Idea Pool and enters Incubation directly. It still needs a Jira ticket and a named owner on day one.
🚫
Killed and Parked ideas are never deleted
Killed ideas stay in Jira with a documented reason. Parked ideas are reviewed every 90 days in Strategic Review. A killed idea from 6 months ago may be valid today.
⏱
60-day rule for Incubation
An idea in Incubation for 60+ days without movement is flagged for kill or park in the next Incubation Meeting. Stagnation is a decision.
04 — Classification
How we classify projects
Every idea, project, and Jira ticket is classified into one of four verticals when first captured on Friday. This classification flows through the entire pipeline.
VerticalWhat it includesExamplesDecision Engine priority
ProductsMining operations — assets that produce physical outputVWC Ghana, Australia GoldProfitability (18%) and Risk (16%) weighted highest
ServicesConsulting and VC matchmaking — expertise-driven revenueSOL, SAMI, Xperience Space, Golden CamelExpertise (18%) and Strategic Alignment (18%) highest
FutureNew ventures: Agriculture, Construction, Software, Mining ExpansionAvocados, Residential Kumasi, CronosAI, VWC CongoInnovation (18%) and Scalability (16%) highest
NewIdeas outside existing verticals — classified before formal scoringAnything emerging outside current categoriesBalanced weights — reclassified before scoring
Why different weights per vertical?
A gold processing operation and a software product are not evaluated on the same criteria. A mining asset lives on profitability and risk management. A consulting business lives on expertise and strategic relationships. A new venture lives on innovation and scalability. Applying the same weights to everything produces meaningless scores. The Decision Engine uses vertical-specific weights so that a high score means something appropriate for each type of business.
05 — Cadence
The weekly meeting cadence
Seven meeting types, each with a single clear purpose. No meeting discusses topics that belong to another meeting.
Monday
VUC Weekly Execution Meeting
45 min · Louis, Yasin, Lucas
Purpose
Strategic prioritization and weekly commitment setting for the Top 5 active projects
Flow
  • Review previous week's commitments (Done / Partial / Missed)
  • Set this week's tasks per person per project
  • Eisenhower Matrix — Do First / Schedule / Delegate / Don't Do
  • R&O section if any ticket status changed this week
Why this way
The Top 5 comes from the monthly Decision Engine — priorities are not re-debated every Monday. The Eisenhower Matrix forces each person to distinguish truly urgent+important from what only feels urgent.
Output: Each person loads their Q1 and Q2 tasks to their personal Jira board with the project prefix [SOL], [GC], etc.
Tuesday AM
VWC Weekly Objectives Meeting
30 min · Louis, Yasin, Lucas, Emmanuel
Purpose
Review Ghana operations KPIs and set pillar objectives across three time horizons
Flow
  • Previous week: committed vs delivered per pillar
  • Ghana Pulse: Emmanuel reports 8 KPIs live (Kg Mined, Processed, Gold Produced, g/t, Operating Days, Avg Hours, Weekly Cost, Main Issue)
  • Four Pillars: this week / this month / 90-day sprint
  • Cross-pillar actions and blockers
Why this way
VWC is operationally separate from VUC. The KPI panel is a forcing function — "Pending" stays visible in the PDF and JSON, making reporting gaps impossible to ignore. Three time horizons prevent the meeting from becoming only tactical.
Output: Pillar objectives in VWC Jira board with owners and deadlines. KPIs logged. Cross-pillar dependencies made explicit.
Tuesday PM
Pipeline Review
45 min · Louis, Yasin, Lucas
Purpose
Evaluate ideas in Idea Pool and Under Review. Move, park, or kill. No presentations — fast decisions.
Flow
  • Idea Pool: check 4 requirements (owner, written urgency, one-line description, strategic fit reason)
  • All 4 met → Under Review. Missing any → return to owner.
  • Under Review: decide → Incubation / Park / Kill (reason required for last two)
Why this way
Ideas without the 4 minimum requirements are sent back before they waste group discussion time. The written reason for Park and Kill creates accountability and historical record.
Output: Jira statuses updated. Ideas moving to Incubation added to next bi-weekly backlog. Summary posted to Teams #pipeline-updates.
Wednesday
No meetings — deep work day
Intentional buffer
Why this way
Monday and Tuesday are meeting-heavy. Without a protected day, the team has no uninterrupted time to execute what was just committed. Wednesday is the day you actually do the work planned on Monday.
Thursday
Async Status Update
No meeting · each owner posts by noon
Format
Three parts: Progress / Blocker / Next step. Link to Jira ticket required. Posted in Teams #active-execution.
Why this way
A 30-minute status meeting every Thursday consumes 26 hours of team time per year and produces nothing that a 3-line written update cannot produce better.
Friday
Ideas Session
30 min · Louis, Yasin, Lucas
Purpose
Protected space to capture ideas without evaluation pressure. No decisions — only capture.
Flow
  • Any idea can be proposed — totally open, any vertical
  • Owner presents, others ask questions only (no killing ideas in this session)
  • Every idea gets a Jira ticket before the session closes: name, owner, urgency, vertical
Why this way
Mixing ideation with evaluation kills creativity. When people know their idea will be immediately judged, they self-censor. The two roles — capture and evaluate — must be in different rooms on different days.
Output: All ideas in Jira. JSON exported and loaded into Tuesday's Pipeline Review. Ideas with MVP go directly to Incubation.
Bi-weekly
Incubation Meeting
45 min · Louis, Yasin, Lucas · Alternating Tuesdays
Purpose
Evaluate ideas in Incubation using a structured presentation and scoring framework.
Flow
  • Backlog review: what is in incubation, how long, any 60-day zombies
  • 5-minute presentation: What / Why now / What we did / What we need / Main risk
  • Score: Strategic Fit (1-5) × Feasibility Now (1-5)
  • Decision: Approve / Continue / Park / Kill (reason required)
Why this way
The Incubation Meeting is the bridge between informal ideas and formal evaluation. Before this system, ideas with working MVPs floated without ever entering the Decision Engine. The scoring matrix gives founders a shared language to evaluate fit and feasibility without extended debate.
Output: Approved ideas added to Decision Engine before next Strategic Review. Park/Kill reasons in Jira. JSON exported for next bi-weekly.
Monthly
Monthly Strategic Review
60 min · Louis, Yasin, Lucas · Last Monday of the month
Purpose
Update the Decision Engine, review the full portfolio, set the new Top 5, review all R&O tickets.
Flow
  • Portfolio: executing / completed / killed this month
  • Quick Kill Filter: eliminate non-viable projects before scoring
  • Decision Engine: new scores, new projects, updated names
  • New Top 5: Louis proposes, the three validate
  • R&O full review: all open tickets regardless of status change
Why this way
Scores updated once a month only — not ad hoc. This prevents individual founders from changing priorities between meetings without group validation. Louis proposes the Top 5 because he holds the strategic vision. The other two validate because execution requires buy-in from everyone delivering it.
Output: New Top 5 loaded into Monday's Priority Stack. R&O statuses updated in Jira. Summary to Teams #decisions.
06 — Tools
Jira setup and ticket types
Jira is the source of truth. Two boards: VUC (shared) and VWC (shared, pillar structure). Each founder also keeps a personal board for individual task management.
IDEA
Idea Pool → Under Review → Incubation
Required: Name, Owner, Vertical (Products / Services / Future / New), Urgency written in ticket, MVP Y/N. An idea without all five fields cannot pass Under Review.
PROJECT
Approved → Active Execution → Completed / Parked / Killed
Required: Vertical tag, subtasks with owners and deadlines before activation. Prefix convention: [SOL], [GC], [VWC], [XS] so each founder can filter their personal board by project.
R&O
Open → Monitoring → Escalated → Actioned → Closed
Required: Type (Risk/Opportunity), Probability, Impact, Owner, Mitigation plan. Only enters weekly agenda if status changed. Full review monthly.
VWC OBJ
Open → In Progress → Closed
Required: Pillar (P1/P2/P3/P4), Owner, Deadline. Lives in the shared VWC board, not in personal boards.
BLOCKER
Open → Resolved
Linked to parent ticket. Founder tagged. Resolved within 48 hours. Urgent blockers go to Teams #blockers immediately.

08 — Operating rules
The rules that keep the system working
These are not preferences. Each one exists because its absence caused a real problem before this system was built.
📋
No agenda = no meeting
Every meeting requires an agenda posted in Teams 24 hours before. No agenda means cancelled or rescheduled.
🛡️
Monday is for active execution only
Zero new ideas. Zero VWC topics. Zero pipeline discussion. This separation is what makes execution possible.
🌍
VWC has its own lane
VWC objectives discussed Tuesday morning only. Mining operations never enter VUC pipeline discussions.
🔇
R&O: no status change = no discussion
R&O only enters the Monday agenda if a Jira ticket status changed that week. If nothing changed, the section is skipped entirely.
📊
Decision Engine updated monthly only
Scores are set in Strategic Review — not ad hoc. This prevents individual founders from changing priorities without group validation.
⚡
Founder approvals get Jira tickets
If a task requires founder approval, it gets a Jira ticket with a deadline. Not chased over Teams chat.
✍️
Urgency must be written
Urgency stated verbally does not move a ticket. It must be written in Jira. Unwritten urgency is not urgency.
09 — Implementation
Getting started — first two weeks
Run a catch-up session before the first official Monday to bring the project registry to its real current state.
1
Catch-up session (before first Monday)
90-minute session with Louis, Yasin, and Lucas. Map every current project to its real pipeline stage. Add them to Jira with vertical tags and named owners. This is the baseline — without it, the Monday tool has nothing to pull from.
Use the Monthly Strategic Review tool to score existing projects and generate the first Top 5.
2
Set up Microsoft Teams channels
Create the 6 channels: #ideas-vuc, #active-execution, #pipeline-updates, #vwc-updates, #decisions, #blockers. Pin the GitHub links to each relevant channel.
3
Set up the VWC Jira board
Create a shared VWC board with four swimlanes (P1/P2/P3/P4). Add Emmanuel as a board member. Each pillar owner manages their own lane. VUC founders keep their personal boards unchanged.
4
First Monday — open the Priority Stack
Open VUC_Weekly_Priority_Stack.html. Click "Start fresh" in Tab 0 (first week only). Load the Top 5 from the catch-up session. Each person writes their commitments. Run the Eisenhower Matrix. Export JSON and PDF.
5
First Friday — open the Ideas Session
Open VUC_Ideas_Session.html. Any idea proposed gets captured with name, owner, urgency, and vertical before the session closes. Export JSON for Tuesday's Pipeline Review.
Remind the team: no idea is killed in this session. Judgment happens on Tuesday.
6
Second Monday — close the loop
Open Priority Stack. In Tab 0, paste last week's JSON. Review what each person committed vs delivered: Done / Partial / Missed. Partial and Missed carry forward automatically. This is where the system proves its value — nothing gets forgotten.

How to update the tools
Changes made during a meeting (writing tasks, dragging to Eisenhower quadrants) are session-only — they live in your browser while the tab is open. To preserve a meeting's output, use "Print / PDF" and "Export JSON" at the end of each session. The JSON is pasted into the following week's tool to carry forward commitments. Structural changes to the tools (updating the Top 5, renaming projects) require editing the HTML file and uploading the new version to GitHub. This happens once a month after Strategic Review.
